# SH-4 MFA Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`).

**Goal:** TOTP MFA via Supabase Auth; admins + sensitive actions require AAL2; full web enroll/challenge/manage UX. (Spec: `docs/superpowers/specs/2026-06-22-sh4-mfa-design.md` — authoritative.)

**Architecture:** Supabase native MFA (no custom crypto). Server reads the JWT `aal` claim and gates sensitive routes with `requireAAL2`; web drives enroll/challenge/manage via `supabase.auth.mfa.*`.

**Tech Stack:** Supabase Auth MFA, Node 22/Express/TS-ESM, React 18 + Vite, Vitest + Supertest + Testing Library.

## Global Constraints
- **Supabase native MFA only** — no custom TOTP/crypto. `aal` claim is the source of truth (aal1=password, aal2=MFA).
- **Scope:** AAL2 required on admin/sensitive routes only (not all users). Reuse SH-3's `AuditRecorder` for `mfa.required` events.
- **ESM `.js`** specifiers. Web client is `web/src/supabase.ts` (`export const supabase`).
- Baseline green: **253 server + 78 web + 21 integration**. Commit per task; do NOT push.
- No DB migration expected (MFA lives in Supabase-managed `auth` schema).

### Task 2: Server AAL2 enforcement

**Files:** Modify `server/src/middleware/require-auth.ts` (attach `req.aal`); Create `server/src/middleware/require-aal2.ts`; Modify `server/src/modules/onboarding/routes.ts` + `server/src/routes/audit.ts` (+ `app.ts` wiring as needed) to apply `requireAAL2` on sensitive routes; Tests: `server/test/require-aal2.test.ts`, extend `server/test/require-auth.test.ts`.

**Interfaces:** Produces `requireAAL2(opts?: { audit?: AuditRecorder }): RequestHandler`; `req.aal: 'aal1'|'aal2'|null`.

- [ ] **Step 1: Write failing tests.**
  - `require-auth`: given a token whose decoded payload has `aal:'aal2'`, `req.aal === 'aal2'`; `aal:'aal1'` → `'aal1'`; missing → `null`. (Build a token as `header.<base64url(JSON payload)>.<sig>`; the middleware decodes the payload only, so the signature can be a dummy because `getUser` is stubbed to return a user.)
  - `require-aal2`: aal1 request → 403 with body `{ code: 'mfa_required' }` AND the injected audit recorder received `action:'mfa.required'`; aal2 request → `next()` called (200 from a downstream handler).
- [ ] **Step 2: RED.** `npm run test --workspace server -- require-aal2 require-auth`
- [ ] **Step 3: Implement.**
  - In `require-auth.ts`: after `getUser` succeeds, parse the token's middle segment: `const payload = JSON.parse(Buffer.from(token.split('.')[1] ?? '', 'base64url').toString('utf8'))` (guard try/catch → null); set `req.aal = payload?.aal ?? null`. Add `aal?: 'aal1'|'aal2'|null` to the `Express.Request` augmentation.
  - `require-aal2.ts`:
    ```ts
    import type { RequestHandler } from 'express'
    import type { AuditRecorder } from '../services/audit.js'
    export function requireAAL2(opts?: { audit?: AuditRecorder }): RequestHandler {
      return (req, res, next) => {
        if (req.aal === 'aal2') return next()
        void opts?.audit?.record({
          workspaceId: req.workspaceId ?? null, actor: req.user?.id ?? null, action: 'mfa.required',
          metadata: { ip: req.ip, route: req.path, method: req.method, requestId: (req.headers['x-request-id'] as string | undefined) ?? null },
        })
        res.status(403).json({ error: 'MFA required', code: 'mfa_required' })
      }
    }
    ```
- [ ] **Step 4: Apply to sensitive routes.** In `onboarding/routes.ts`, add `requireAAL2({ audit: deps.audit })` after `requireWorkspaceAdmin(...)` on the mutations (`/finish`, `/retry/:id`, and the step writes `/profile`,`/rules`,`/industry`,`/people`) — keep `GET /session` at aal1. In `routes/audit.ts`, add `requireAAL2({ audit: deps.audit })` after the admin gate. Wire `deps.audit` is already present (SH-3).
- [ ] **Step 5: GREEN + full server suite.** `npm run test --workspace server`. Note: existing integration tests authenticate at aal1 (no MFA), so any now-aal2-gated route they exercise will 403 — update those integration flows to either enroll+verify MFA (preferred if feasible) OR adjust which routes are gated. If integration breakage is broad, gate only `/finish`,`/retry/:id`,`/api/audit` (the truly sensitive ones) and leave step-writes at aal1; document the choice. Re-run `npm run test:int --workspace server` and resolve.
- [ ] **Step 6: Commit.** `git add -A && git commit -m "feat(mfa): read aal claim + requireAAL2 on sensitive routes (SH-4)"`
